# Gemini Bridge: приём и разбор payload

**Фаза**: 5
**Статус**: backlog
**Приоритет**: medium
**Зависимости**: phase4-003-edit-card-command, phase4-002-plant-info-command

## Описание

Принять структурированный вывод Gemini (JSON или Markdown), переданный пользователем через Telegram-сообщение, и распознать тип операции: обновление карточки, запись в changelog, создание задачи.

**Зачем**: Пользователь ведёт глубокий анализ в Gemini (с доступом к репозиторию `trava`), затем передаёт результат боту для применения изменений — без ручного редактирования файлов.

**Кто**: editor+

**Что**: Специальный триггер (команда или маркер в тексте) активирует парсер Gemini-payload; бот определяет тип операции и готовит изменения к применению.

## Критерии готовности

- [ ] Определён формат Gemini-payload (JSON-схема или Markdown-шаблон, задокументирован в `docs/spec/`)
- [ ] Команда или маркер для активации режима Gemini Bridge (например `/gemini` или блок ```gemini ... ```)
- [ ] Парсер разбирает payload и возвращает список операций (тип + данные)
- [ ] Нераспознанный payload → user-friendly ошибка с подсказкой по формату
- [ ] Unit-тесты на парсер

## Заметки

- Triгер: Telegram-пересылка или явная команда — решить на этапе дизайна
- Payload может содержать несколько операций сразу (batch)
- Связано с `phase5-002-apply-gemini-changes`
