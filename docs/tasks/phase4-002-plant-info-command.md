# Команда `/info` — просмотр карточки растения

**Фаза**: 4
**Статус**: backlog
**Приоритет**: high
**Зависимости**: нет (plant_repo.read_card уже реализован)

## Описание

Пользователь хочет быстро посмотреть текущую информацию о растении: что за вид, когда куплено, здоровье, последние записи из журнала — прямо в Telegram, без необходимости открывать git-репо или markdown-файл.

**Зачем**: Основной use-case бота — мониторинг. Без возможности посмотреть карточку бот «слепой» — пользователь пишет действия, но не видит текущее состояние.

**Кто**: viewer+ (минимальная роль — любой авторизованный пользователь)

**Что**: Команда `/info [plant_id]` (или по текущему выбранному растению) отображает форматированную сводку из YAML front matter + последние N записей changelog.

## Сценарии использования

### 1. `/info` — по текущему растению из сессии
```
User: /info
Bot:
🌿 Хризалидокарпус желтоватый (Пальма Арека)
━━━━━━━━━━━━━━━━━━━
🪪 ID: areca_01
🧬 Chrysalidocarpus lutescens
📅 Куплено: 2025-12-22 (IKEA)
🏺 Горшок: 17 см, plastic
❤️ Здоровье: 7.5/10
🏷 пальма, после_обрезки, требуется_пересадка

📜 Последние записи:
• 2026-05-15: Проведена радикальная санитарная обрезка...
```

### 2. `/info calathea_01` — явное указание plant_id
Работает без предварительного `/plants` + выбора.

### 3. Через inline-кнопку после `/plants`
В клавиатуре `/plants` добавить кнопку «ℹ️ Инфо» рядом с «✅ Выбрать».

### 4. Через LLM (Фаза 3)
Свободный текст «Расскажи про калатею» → intent `action: info` → тот же handler.

## Требования

### Функциональные
- Команда `/info` без аргументов использует `current_plant_id` из сессии
- Команда `/info <query>` ищет растение через `plant_repo.find(query)` (поддержка fuzzy-поиска по имени)
- Если растение не найдено → понятное сообщение + предложить `/plants`
- Если сессия пустая и нет аргумента → «Выбери растение через /plants»
- Форматирование: HTML (Telegram parse_mode), компактно, читаемо на телефоне
- Changelog: последние 3–5 записей (обрезать длинные > 120 символов)
- Скрытые/пустые поля не отображать

### Нефункциональные
- Минимальная роль: `viewer`
- Не требует git-операций (чтение с диска)
- Ответ < 1 секунды (всё локально, без LLM)

## Формат ответа бота

```
🌿 <b>{common_name}</b>
━━━━━━━━━━━━━━━━━━━
🪪 ID: <code>{id}</code>
🧬 {botanical_name}
📅 Куплено: {purchase_date} ({purchase_location})
🏺 Горшок: {pot_size_cm} см, {pot_type}
💧 Последний полив: {из changelog, если есть}
❤️ Здоровье: {health_score}/10
🏷 {tags через запятую}

📜 <b>Последние записи:</b>
• {date}: {entry_text_truncated}
...
```

## Критерии готовности
- [ ] Handler `/info` зарегистрирован в `bot/handlers/`
- [ ] Работает без аргумента (текущее растение из сессии)
- [ ] Работает с аргументом (fuzzy-поиск через `plant_repo.find`)
- [ ] Changelog парсится и выводятся последние 3–5 записей
- [ ] Пустые поля не отображаются
- [ ] Доступ: `viewer+`
- [ ] Ответ при отсутствии растения/сессии — понятный fallback

## Заметки

### Что уже готово
- `plant_repo.read_card(repo_path, plant_id)` → `(yaml_dict, body_text)`
- `plant_repo.find(repo_path, query)` → fuzzy-поиск по id/имени
- `md_parser.read(path)` → парсинг YAML + body
- `sessions_repo.get_current_plant(db, tg_id)` → текущее растение
- Роутер `/plants` с inline-клавиатурой

### Нужно реализовать
- Форматтер: `yaml_dict + body → telegram HTML-сообщение`
- Парсинг changelog-записей из body (regex по `- **YYYY-MM-DD**:`)
- Handler + регистрация команды
- (Опционально) Кнопка «ℹ️» в `/plants` inline keyboard

### Интеграция с LLM (Фаза 3)
В `intent_schema.json` добавить action `info` → вызов того же форматтера.
